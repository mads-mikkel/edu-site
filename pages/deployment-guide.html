Deployment Guide

This guide walks you through every step needed to get your site running on the school server (3kant-server) using Gitea, Docker, and Gitea Actions.

End result: every time you push code to your main branch, the server automatically builds a new Docker image of your site, pushes it to the school's container registry, and restarts your container — so your live site updates automatically.

Prerequisites

  • A Gitea account on http://3kant-server.cv.local:1103/
  • A repo on that Gitea instance containing your site files, a dockerfile, a docker-compose.yaml, and a .gitea/workflows/deploy.yml
  • Your site must have at least an index.html

Overview of the Pipeline

You push to main
       │
       ▼
Gitea Actions runner picks up the workflow
       │
       ├─► Builds a Docker image of your site
       │
       ├─► Pushes the image to the school's container registry
       │         (3kant-server.cv.local:1103)
       │
       └─► Runs docker compose up → your site is live on your port

Step 1 — Create or Fork the Repo in Gitea

  1. Open http://3kant-server.cv.local:1103/ and log in.
  2. Either:
    • Fork an existing repo (e.g. mara/edu-site): click the Fork button on the repo page, or
    • Create a new repo: click the + icon → New Repository. Give it a name (e.g. my-website) and click Create Repository.
  3. If you created a new repo, clone it to your machine and add your site files.

Step 2 — Add the Required Files

Your repo needs three infrastructure files. If you forked mara/edu-site they are already there — but you must edit them as described in Steps 3 and 4.

dockerfile

No changes needed. This packages your site into an nginx image.

FROM nginx:latest

COPY . /usr/share/nginx/html

EXPOSE 80

CMD ["nginx", "-g", "daemon off;"]

docker-compose.yaml

Tells Docker how to run your container on the server. You must customize this — see Step 3.

.gitea/workflows/deploy.yml

Defines the automated pipeline. You must customize this — see Step 4.

Step 3 — Customize docker-compose.yaml

Open docker-compose.yaml. It looks like this:

services:
  edu-site:
    image: 3kant-server.cv.local:1103/mara/edu-site:latest
    container_name: mara-edu-container
    ports:
      - "1104:80"
    restart: unless-stopped

Change three things so your container does not clash with anyone else's:

Field What to change Example
image Replace mara/edu-site with your Gitea username and your repo name 3kant-server.cv.local:1103/anna/my-website:latest
container_name Use your own unique name anna-my-website-container
ports Replace 1104 with the port number assigned to you "2050:80"

The port on the left of the : is the public port your site will be reachable on. The 80 on the right must stay as-is — that is nginx's internal port.

Example after editing:

services:
  my-website:
    image: 3kant-server.cv.local:1103/anna/my-website:latest
    container_name: anna-my-website-container
    ports:
      - "2050:80"
    restart: unless-stopped

Step 4 — Customize .gitea/workflows/deploy.yml

Open .gitea/workflows/deploy.yml. It looks like this:

name: Build and Deploy

on:
  push:
    branches:
      - main

jobs:
  build-and-deploy:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Log in to Gitea container registry
        uses: docker/login-action@v3
        with:
          registry: 3kant-server.cv.local:1103
          username: ${{ secrets.REGISTRY_USERNAME }}
          password: ${{ secrets.REGISTRY_PASSWORD }}

      - name: Build and push Docker image
        uses: docker/build-push-action@v5
        with:
          context: .
          file: ./dockerfile
          push: true
          tags: 3kant-server.cv.local:1103/mara/edu-site:latest

      - name: Deploy — pull new image and restart container
        run: |
          docker compose pull
          docker compose up -d

Change one line — the tags: value under Build and push Docker image. Set it to match the image: value you used in docker-compose.yaml: 

          tags: 3kant-server.cv.local:1103/anna/my-website:latest

⚠️ Important: the tags: value in the workflow and the image: value in docker-compose.yaml must be identical. If they differ, the runner will push the image under one name but try to run a different one.

Step 5 — Add Secrets to Your Gitea Repo

Your workflow needs your Gitea credentials to push to the registry. You store them as secrets — encrypted values that Actions can read but that are never visible in logs or code.

  1. In your repo on Gitea, click Settings (top-right of the repo page).
  2. In the left sidebar, click Actions → Secrets.
  3. Click Add Secret and add the following two secrets:
Secret name Value
REGISTRY_USERNAME Your Gitea username (e.g. anna)
REGISTRY_PASSWORD Your Gitea password

After adding both, the secrets page should list both names. The values are hidden — that is expected.

⚠️ Never put your password directly in the workflow file. Always use secrets.

Step 6 — Push to Main and Watch It Deploy

  1. Commit and push all your changes (the updated docker-compose.yaml, .gitea/workflows/deploy.yml, and your site files) to the main branch.
  2. In your repo on Gitea, click the Actions tab. A workflow run should appear within a few seconds.
  3. Click on the run to expand it and watch the steps execute:
    • Checkout code — the runner fetches your repo
    • Log in to Gitea container registry — authenticates using your secrets
    • Build and push Docker image — builds the nginx image and pushes it to the registry
    • Deploy — pulls the new image and starts your container

A green ✔ next to every step means success.

Step 7 — Verify Your Site is Live

Open a browser and go to:

http://3kant-server.cv.local:<your-port>/

Replace <your-port> with the port you chose in Step 3 (e.g. http://3kant-server.cv.local:2050/).

You should see your index.html. 🎉

From now on, every push to main triggers an automatic redeploy — no manual steps needed.

How Subsequent Deploys Work

After the first deploy, the server is already running your container. When you push again:

  1. The runner builds a new Docker image with your latest code.
  2. It pushes the new image to the registry, overwriting :latest.
  3. docker compose pull downloads the new image to the server.
  4. docker compose up -d recreates the container using the new image — automatically.

Troubleshooting

No workflow run appears after pushing

  • Make sure your workflow file is saved at exactly .gitea/workflows/deploy.yml (the .gitea folder must be at the root of the repo).
  • Check that you pushed to the main branch, not master or another branch.

"Log in to Gitea container registry" fails

  • Double-check that both secrets are set in Settings → Actions → Secrets of your repo.
  • Make sure the username and password match your Gitea login exactly (case-sensitive).

"Build and push Docker image" fails with denied or unauthorized

  • The tags: value must start with your own Gitea username. You cannot push to another user's namespace.

"Deploy" fails with port is already allocated

  • The port you chose is already used by another container. Pick a different port, update docker-compose.yaml, and push again.

The site shows nginx's default page instead of my content

  • Make sure index.html is in the root of your repo, not inside a subfolder. The dockerfile copies everything to /usr/share/nginx/html and nginx looks for index.html there.

I visit my port but get "connection refused"

  • The container may have exited after starting. Check the log of the Deploy step in your workflow run for error output from Docker.
